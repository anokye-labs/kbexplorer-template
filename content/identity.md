The identity system (`src/engine/identity.ts`) solves a fundamental problem: the same real-world entity — a source file, a component, a concept — can appear as multiple nodes produced by different [providers](providers-overview). Without stable identities, the [graph engine](graph-engine) would treat `urn:file:src/engine/graph.ts` from the [files provider](files-provider) and the authored `graph-engine` content node as completely unrelated things.

## How Identity Works

Each `KBNode` receives an `identity` field — a URN string generated by `assignIdentity()` based on the node's `source.type`. The function is a simple switch that maps source types to URN namespaces:

```typescript
export function assignIdentity(node: KBNode): string | undefined {
  switch (node.source.type) {
    case 'file':         return `urn:file:${node.source.path}`;
    case 'authored':     return `urn:content:${node.id}`;
    case 'readme':       return 'urn:content:readme';
    case 'issue':        return `urn:issue:${node.source.number}`;
    case 'pull_request': return `urn:pr:${node.source.number}`;
    case 'commit':       return `urn:commit:${node.source.sha}`;
    case 'section':      return undefined; // sections don't get identities
  }
}
```

Section nodes intentionally return `undefined` — they are sub-parts of a parent and don't represent independent entities.

## Identity Matching

`shareIdentity(a, b)` checks whether two nodes represent the same entity by comparing their URN strings. This is used by the view system to merge representations — if a file node and a content node share an identity, the graph can display them as facets of the same concept. The full design rationale is in the [multi-layer identity spec](multi-layer-identity), implemented in [#47](https://github.com/anokye-labs/kbexplorer-template/issues/47) alongside the provider system (commit `930f94c`).

## The Identity Index

`buildIdentityIndex()` creates a `Map<string, string[]>` from URN to all node IDs sharing that identity. This enables the [parser](parser) to merge information from different sources and the view system to cross-reference layers.

## Why URNs

URNs were chosen because file paths change during refactoring, issue numbers are repo-specific, and content IDs are arbitrary. URNs provide a stable, namespaced identifier that survives restructuring. The `file:` field in `nodemap.yaml` entries creates identity links between authored content and source files — see the [node mapping](node-mapping) system. The Visual Identity System ([#9](https://github.com/anokye-labs/kbexplorer-template/issues/9)) ensures these identities render consistently across all seven display surfaces.

## Integration Points

The identity system is called by:
- The [parser](parser) during `parseMarkdownFile()` — sets identity on authored nodes
- The [authored provider](authored-provider) — ensures `urn:content:` prefix
- The [files provider](files-provider) — assigns `urn:file:` identities
- The [work provider](work-provider) — assigns `urn:issue:` and `urn:pr:` identities
- The [local loader](local-loader) — identity assignment for manifest-loaded nodes

All URN formats are defined in the [type system](type-system) and documented in the [multi-layer identity spec](multi-layer-identity).
